# 四川非遗文旅数字导览小程序 - 项目申报书

---

## 二、项目核心概述

**项目名称**：基于LBS与云数据库的四川非遗文旅导览系统

**项目定位**：微信小程序（软硬件结合体）——面向文旅场景的智能导览与互动平台

本项目是一款基于uni-app与微信云开发的四川非物质文化遗产文旅数字导览小程序，为用户提供非遗项目展示、景点导览、打卡记录、路线规划等一站式文旅服务。系统采用NoSQL云数据库存储非遗、景点、用户行为等多维度数据，通过Haversine算法实现500米地理围栏打卡验证，融合腾讯地图SDK与百度TTS语音合成技术，构建完整的"数据管理-地理位置服务-语音交互"三维一体架构。数据结构在项目中承担核心作用：云数据库集合构成数据层基石，地理位置算法支撑打卡验证业务，Polyline解码算法实现路线可视化，真正实现非遗文化的数字化传播与活态传承。

## 四、用户核心痛点分析

### 1. 痛点一：非遗文化信息分散，获取成本高

传统非遗文化传播依赖线下场馆或书籍资料，用户难以便捷获取四川各地非遗项目的详细信息（历史渊源、技艺特色、地理位置等）。本项目通过云数据库集中管理非遗数据，支持多字段模糊搜索（正则OR查询），用户输入关键词即可快速定位目标非遗项目。

### 2. 痛点二：文旅打卡缺乏可信验真机制

市面打卡应用缺乏地理位置真实性验证，用户可随意上传照片"云打卡"。本项目通过Haversine球面距离公式计算用户与打卡点的实际距离（500米围栏），要求用户必须在现场方可完成打卡，确保打卡记录的真实可信。

### 3. 痛点三：景区导览缺乏智能化路线规划

散客游览多景点时缺乏路线规划能力，依赖地图App需手动输入多个目的地，操作繁琐。本项目整合腾讯地图SDK，支持驾车/公交/步行多模式路线规划，通过Polyline解码还原完整路径，在地图上直观展示途经点与路线。

## 七、项目核心设计思路及如何赚钱

### 7.1 核心设计原则

1. **简洁易用**：小程序即用即走，用户无需下载安装，扫码即用
2. **算法高效**：云数据库原子操作防并发，Haversine算法O(1)复杂度，分页查询O(n)可控
3. **结构清晰**：MVC架构分离，工具函数模块化（cloudCall/storage/auth/mapService）
4. **贴合课程**：线性表（浏览历史）→ 云数据库数组；树结构应用→ 分类筛选；图结构→ 地理位置网络
5. **容错健壮**：离线降级策略（云端失败用本地缓存兜底）

### 功能拆解逻辑

将项目拆分为 **6个核心模块**，分别为：

| 模块 | 功能描述 | 数据结构依赖 |
|------|----------|--------------|
| 非遗模块 | 非遗列表展示、详情浏览、语音导览 | heritage集合、正则搜索 |
| 景点模块 | 景点列表、地图展示、导航唤起 | scenic集合（含经纬度） |
| 打卡模块 | 位置验证、照片上传、打卡记录 | checkin集合、Haversine算法 |
| 收藏模块 | 收藏添加/取消、收藏列表展示 | user_favorites集合 |
| 路线模块 | 多途经点规划、路线展示、导航 | routes集合、Polyline解码 |
| 用户模块 | 登录鉴权、信息管理、历史记录 | users/browse_history集合 |

### 核心解决思路

针对用户痛点，通过选用合适的数据结构实现以下功能：

| 痛点 | 数据结构解决方案 | 课程知识点 |
|------|------------------|-----------|
| 信息分散难检索 | NoSQL云数据库 + 多字段正则OR查询 | 查找算法（正则匹配） |
| 打卡缺乏验真 | Haversine球面距离算法（500m围栏） | 图论/几何计算 |
| 路线规划繁琐 | 腾讯地图SDK + Polyline解码 | 线性表（坐标序列） |
| 浏览历史混乱 | 本地存储TTL缓存 + 云端同步 | 栈/队列思想 |
| 收藏管理分散 | 云数据库user_favorites集合 | 集合（Set）去重 |

## 九、核心用到的数据结构及选型理由

| 数据结构名称 | 具体应用场景 | 选型核心理由 | 核心操作实现 |
|-------------|-------------|-------------|-------------|
| **NoSQL文档数据库集合** | 非遗/景点/打卡/收藏等9个数据集合的CRUD操作 | 支持灵活字段、原子操作、地理位置查询，云函数自动注入openid实现权限隔离 | `db.collection().add()` / `where()` / `update()` / `remove()`，时间复杂度O(n)（依赖索引可优化） |
| **Haversine球面距离算法** | 用户与打卡点的500米地理围栏验证 | 基于球面三角学公式计算地球表面两点距离，避免平面投影失真，适用于GPS坐标验证场景 | `calculateDistance(lat1,lng1,lat2,lng2)` → `a=sin²(Δlat/2)+cos(lat1)×cos(lat2)×sin²(Δlng/2)`，时间复杂度O(1) |
| **Polyline Delta解码算法** | 腾讯地图路线规划返回的压缩坐标串解码 | 腾讯地图返回分号分隔的增量序列，每个值除以1,000,000后与前一点累加，还原绝对坐标序列用于路线绘制 | 遍历编码串 → 累加增量 → 归一化(÷1,000,000) → 存储坐标对，时间复杂度O(n) |
| **Skip-Limit分页查询** | 非遗/景点/打卡等列表的无限滚动加载 | 云数据库不支持游标分页，采用skip+limit固定偏移量实现分页，控制每次返回数据量 | `skip((page-1)×pageSize).limit(pageSize)`，时间复杂度O(n×skip) |
| **防抖（Debounce）算法** | 搜索框输入防抖，避免频繁触发云数据库查询 | 基于setTimeout/clearTimeout实现，500ms内连续输入仅执行最后一次查询，降低云函数调用次数 | `clearTimeout(timer)` → `setTimeout(doSearch, 500)`，时间复杂度O(1) |
| **TTL本地缓存** | 搜索历史、认证信息、路线缓存的过期管理 | 存储格式包含value/timestamp/expire字段，读写时检查timestamp判断是否过期，避免内存泄漏 | `isExpired() → Date.now() - timestamp > expire`，空间复杂度O(k)，k=缓存条数 |
| **Set集合（去重）** | 分类筛选标签去重、搜索历史去重 | 使用JavaScript原生Set自动去重，保证筛选标签无重复项，搜索历史保持唯一性 | `new Set(array)` → `[...set]`，时间复杂度O(n) |
| **状态机模式** | 用户登录状态流转（游客→已登录无资料→已登录有资料） | 三态单向流转设计，配合7天Session有效期，实现安全的用户认证闭环 | `silentLogin()` → `authorizeLogin()` → `ensureLogin()`，状态转换时间复杂度O(1) |

**补充说明**：

| 知识点类别 | 具体应用 | 复杂度优势 |
|-----------|---------|-----------|
| **查找算法** | 多字段正则OR查询（name/brief/category/location/tags） | 正则匹配O(m×n)，m=字段数，n=文档数，配合索引可优化 |
| **线性表应用** | 坐标序列Polyline解码、浏览历史列表、收藏列表 | 顺序表O(1)随机访问，O(n)插入，栈思想用于历史记录回退 |
| **图论应用** | Haversine距离计算（构建地理围栏图）、路线节点网络 | O(1)单点距离计算，路径规划依赖地图SDK内置算法 |
| **排序算法** | 类别标签按预设顺序排列（传统音乐→传统舞蹈→...） | 自定义排序O(n log n)，稳定排序保证类别顺序固定 |

## 十二、备注与补充说明

### 技术亮点

1. **离线降级策略**：云函数调用失败时自动降级到本地mock数据（首页）或本地缓存（打卡、路线），保证核心功能可用性

2. **隐私优先设计**：地图默认显示成都中心点，不自动申请GPS权限，仅用户主动点击"我的位置"时才触发定位

3. **原子操作防并发**：浏览量统计使用`db.command.inc(1)`原子自增，避免多用户同时访问时的计数不准问题

4. **图片批量处理**：云存储临时链接转换采用每批10个的批次处理，配合Promise.race超时保护，避免大图集加载阻塞

### 课程知识点映射

| 课程章节 | 项目应用 | 实现文件 |
